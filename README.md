## 👥 共同開発の方法について（方針検討中）

現在、このプロジェクトをどのように共同開発していくか（データの共有方法）を検討中です。
一長一短あるため、ひとまず**「Unity Packageでの共有」**と**「Gitでの完全管理」**のどちらにも対応できるように、このREADMEにそれぞれの長所・短所と手順を残しています。

メンバーの状況に合わせてどちらにするか決めましょう！

---

### 📦 プランA: `.unitypackage` でやり取りする
特定のシーンやアセットだけを書き出して、Discordやドライブ等で送り合う方法です。

*   **長所（メリット）：**
    *   Gitの難しい操作（クローンやコミットなど）を覚える必要がない。
    *   「自分が作ったこの部品だけを相手に渡したい」という時にサクッと動かせる。
*   **短所（デメリット）：**
    *   同じファイルを2人が同時に編集すると、インポートした時に古いデータが上書きされて消える（データ衝突の危険大）。
    *   プロジェクト内でファイルを「削除」した情報が相手に同期されない（ゴミファイルが残りやすい）。
    *   ゲーム全体の共通設定（Project Settings）を共有しにくい。
*   **この方法で行く場合のルール：**
    *   作業するシーン（Sceneファイル）や担当するスクリプトを完全に分けること。
    *   同じファイルを同時に編集しないように、事前にチャット等で声を掛け合うこと。

---

### 🌐 プランB: Git / GitHub でプロジェクトごと完全管理する（推奨）
このGitHubリポジトリをベースに、全員が同じプロジェクトを同期しながら開発する方法です。

*   **長所（メリット）：**
    *   誰がどこを書き換えたかの履歴がすべて残り、いつでも過去の状態に戻せる。
    *   同じファイルを編集してしまっても、Git上で変更をマージ（合体）させることができる。
    *   アセットの追加だけでなく「削除」も全員に正しく同期される。
    *   Project Settingsも含めて、ゲーム丸ごと完全に同じ状態を保てる。
*   **短所（デメリット）：**
    *   Git（GitHub DesktopやSourceTreeなど）の基本操作を全員が覚える必要がある。
    *   Unity特有のコンフリクト（データの衝突）が起きた際、解消に少しコツがいる（特にSceneファイルやPrefabファイル）。
*   **この方法で行く場合の注意点：**
    *   不要な自動生成ファイルをアップロードしないよう、`.gitignore` を正しく設定すること。
    *   容量の大きい大きな3Dモデルや音源を扱う場合は、必要に応じて `Git LFS` の導入を検討すること。

## 🛠️ それぞれの使い始め方

### プランA（パッケージ）で進める場合
1.  **書き出し側：** UnityのProjectウィンドウで渡したいシーンやフォルダを右クリック ＞ `Export Package`（*Include dependencies* にチェックを入れる）。
2.  出来上がった `.unitypackage` を配布する。
3.  **受け取り側：** 自分のUnityプロジェクトを開いた状態で、受け取ったパッケージファイルをダブルクリックしてインポートする。

### プランB（Git管理）で進める場合
1.  このリポジトリを各自のPCに `Clone（クローン）` する。
2.  各自、Unity Hubからクローンしたフォルダを追加してプロジェクトを開く。
3.  編集後は、意味のある単位で `Commit` ＆ `Push` を行い、作業前に必ず `Pull` をして最新状態に更新する。
